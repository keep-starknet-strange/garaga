# RSA-2048 Cairo Plan

## Goal

Add a low-level RSA-2048 verifier under `garaga::signatures` that checks:

- `signature^65537 mod N == expected_message_rep`
- with the exact RNS/CRT witness rules from [`rns_crt_cairo_exact_demo.py`](./rns_crt_cairo_exact_demo.py)
- with Python witness and calldata generation integrated into the existing signature test flow

This verifier is intentionally arithmetic-only. Hashing and PKCS#1 message encoding remain outside the v1 Cairo API.

## Scope Freeze

- MVP verifies only the arithmetic path from the demo.
- Keep the public exponent fixed to `e = 65537`.
- Do not implement generic exponents in v1.
- Do not implement PSS or PKCS#1 v1.5 encoding in v1.
- Do not expose the API as if it validates a full RSA signature scheme.
- Return `bool` on invalid witness or input data; do not panic from the public verifier.

## Circuit Evaluation Budget

The verifier budget is:

- Encoding checks: `37` encoded values x `11` channels = `407` Horner evaluations
- Relation checks: `17` reductions x `11` channels x `1` relation evaluation = `187` evaluations
- Total: `594` circuit evaluations

Breakdown of the `37` encoded values:

- `3` public values: modulus, signature, expected message
- `34` witness values: `17` quotients and `17` remainders

This budget requires a dedicated generated relation circuit. Do not assemble each channel relation from `mul_mod_p` and `add_mod_p` one-op helpers.

## Source Of Truth

Reuse the exact constants and witness rules from [`rns_crt_cairo_exact_demo.py`](./rns_crt_cairo_exact_demo.py):

- parameter block: `LIMB_BITS`, `BASE`, `N_LIMBS`, `RSA_BITS`, `TOP_LIMB_BITS`, `TOP_LIMB_MAX`, `DELTA_ABS_BOUND`, `CHANNEL_MODULI`
- encoding logic: `split_limbs_2048`, `join_limbs_2048`, `EncodedValue.from_int`
- RNS checks: `CairoRNSContext.residues_from_limbs`, `check_encoding`, `cairo_style_relation_holds`, `check_exact_mul_div_mod`
- witness generation: `witness_mul_div_mod`
- exponentiation trace shape: `verify_rsa_signature_with_exact_checks`

Important constants from the demo:

- radix base: `2^96`
- width: `22` limbs
- top-limb bound: `32` bits
- channel count: `11`
- exactness bound: `DELTA_ABS_BOUND = 2^(4097) + 2^2048`
- exponentiation path for `65537`: `16` squarings + `1` multiply = `17` checked reductions

Do not change the channel moduli in the first PR. Paste them directly from the demo and split them into four `u96` words with the existing Python helpers.

Precompute the Cairo-side chunk steps `2^384 mod p_i` once and keep them as verifier constants.

- Keep reusable modular arithmetic circuits in `hydra/garaga/precompiled_circuits/compilable_circuits/rsa.py` and emit them into `src/src/circuits/rsa.cairo` through `make rewrite`.

## Cairo Data Model

Use the six-word chunk representation throughout the verifier:

```cairo
#[derive(Copy, Drop, Debug, PartialEq)]
pub struct RSA2048Chunks {
    pub w0: u384,
    pub w1: u384,
    pub w2: u384,
    pub w3: u384,
    pub w4: u384,
    pub w5: u384, // top chunk, must be < 2^128
}

#[derive(Drop, Debug, PartialEq)]
pub struct RSA2048EncodedValue {
    pub chunks: RSA2048Chunks,
    pub residues: [u384; 11],
}

#[derive(Drop, Debug, PartialEq)]
pub struct RSA2048PublicKey {
    pub modulus: RSA2048EncodedValue,
}

#[derive(Drop, Debug, PartialEq)]
pub struct RSA2048Signature {
    pub signature: RSA2048EncodedValue,
    pub expected_message: RSA2048EncodedValue,
}

#[derive(Drop, Debug, PartialEq)]
pub struct RSA2048ReductionWitness {
    pub quotient: RSA2048EncodedValue,
    pub remainder: RSA2048EncodedValue,
}

#[derive(Drop, Debug, PartialEq)]
pub struct RSA2048SignatureWithHint {
    pub signature: RSA2048Signature,
    pub reductions: [RSA2048ReductionWitness; 17],
}
```

Exact felt layout:

- one encoded value = `6 * 4 + 11 * 4 = 68` felts
- one reduction witness = `136` felts
- `RSA2048SignatureWithHint` = `68 + 68 + 17 * 136 = 2448` felts
- full positive sample with public key first = `68 + 2448 = 2516` felts

Because `reductions` is a fixed `[...; 17]`, there is no runtime length check and no hidden length prefix.

### `u384` Comparison

`u96` has `PartialEq`, but it does not provide the ordering support needed for lexicographic comparison. Implement ordering by upcasting each `u96` limb to `u128` with `corelib_imports::bounded_int::upcast`, following the same pattern used by `is_even_u384` in `src/src/basic_field_ops.cairo`.

Implementation rule:

- compare chunks from `w5` down to `w0`
- for each `u384`, compare `limb3`, `limb2`, `limb1`, `limb0`
- upcast each limb to `u128` before `<` or `>`
- exit early on the first unequal limb

Worst case cost is `24` `u128` comparisons, which is negligible relative to circuit evaluation cost.

### Top-Chunk Bound

The top chunk must satisfy `< 2^128`.

Implement `top_chunk_is_well_formed(top)` as:

- `top.limb2 == 0`
- `top.limb3 == 0`
- `upcast(top.limb1) <= RSA_TOP_CHUNK_LIMB1_MAX`

Do not declare `RSA_TOP_CHUNK_LIMB1_MAX` as `u96`; the comparison must be done as `u128`.

### Relation Check Circuit

Add a second generated circuit, `RSARelationCheckCircuit`, alongside `RSAEval6ChunksCircuit` and register it as `CircuitID.RSA_RELATION_CHECK` in `hydra/garaga/precompiled_circuits/all_circuits.py`.

Generated signature:

```cairo
pub fn run_RSA_RELATION_CHECK_circuit(
    a: u384, b: u384, q: u384, n: u384, r: u384,
    modulus: core::circuit::CircuitModulus,
) -> (u384,)
```

This circuit computes `a * b - q * n - r` in one evaluation. The caller checks that the returned value is zero:

```cairo
let (diff,) = run_RSA_RELATION_CHECK_circuit(a_i, b_i, q_i, n_i, r_i, channel_mod);
if !diff.is_zero() {
    return false;
}
```

## Encoding Checks

`check_encoding` mirrors the demo over the chunked Cairo shape:

1. `x.chunks.w0` through `x.chunks.w4` are unconstrained `u384`
2. `x.chunks.w5` is checked to be `< 2^128`
3. recompute every channel residue by calling `run_RSA_EVAL_6_CHUNKS_circuit` with `x.chunks`, `get_rsa_chunk_step(i)`, and `get_rsa_channel_modulus(i)`
4. compare the recomputed `u384` residues against `x.residues`

Each encoding check costs `11` Horner circuit evaluations.

## Range Checks And Modulus Validation

The public verifier rejects:

- malformed modulus, signature, or expected-message encodings
- `N <= 1`
- even `N`
- `signature >= N`
- `expected_message >= N`

Implementation details:

- `N > 1`: compare the chunk struct against zero and against the constant `1` with `PartialEq` on `RSA2048Chunks`
- `N` odd: reuse `is_even_u384` from `src/src/basic_field_ops.cairo` on `w0`
- `signature < N`, `expected_message < N`, and `remainder < N`: use `chunks_lt`

If the modulus is held by snapshot, dereference `w0` before calling `is_even_u384`.

Do not add a `q < N` check in v1. The exactness argument only requires `q` to be a well-formed 2048-bit encoded value.

## Reduction-Witness Checking

Use this helper shape:

```cairo
fn reduction_witness_is_valid(
    lhs: @RSA2048EncodedValue,
    rhs: @RSA2048EncodedValue,
    modulus: @RSA2048EncodedValue,
    witness: @RSA2048ReductionWitness,
) -> bool
```

This helper:

1. validates `witness.quotient` with `check_encoding`
2. validates `witness.remainder` with `check_encoding`
3. requires `witness.remainder < modulus`
4. checks every channel relation with `run_RSA_RELATION_CHECK_circuit`

Per channel:

- `a_i = lhs.residues[i]`
- `b_i = rhs.residues[i]`
- `q_i = witness.quotient.residues[i]`
- `n_i = modulus.residues[i]`
- `r_i = witness.remainder.residues[i]`
- call `run_RSA_RELATION_CHECK_circuit(a_i, b_i, q_i, n_i, r_i, channel_mod)`
- require the returned `diff` to satisfy `diff.is_zero()`

Validation policy:

- validate `public_key.modulus`, `signature.signature`, and `signature.expected_message` once at the top level
- do not re-check those public encodings inside every reduction step
- once `acc` comes from a validated remainder produced by an earlier checked reduction, treat it as already encoded

## Hardcoded `65537` Exponent Path

Implement the exact demo path:

1. start with `acc = signature.signature`
2. apply `16` checked squarings
3. apply `1` checked multiply by the base signature
4. compare the final chunks to `expected_message.chunks`

Use the fixed-array witness order directly. Take a snapshot of the input first so the base signature can be reused after `acc` is initialized. Destructure the `[RSA2048ReductionWitness; 17]`, put the first `16` witnesses into a dynamic array for the squaring loop, then process the final multiply explicitly:

```cairo
let hinted_signature = @signature;
let [r0, r1, r2, r3, r4, r5, r6, r7, r8, r9, r10, r11, r12, r13, r14, r15, r16] = *hinted_signature.reductions;
let mut squarings = array![r0, r1, r2, r3, r4, r5, r6, r7, r8, r9, r10, r11, r12, r13, r14, r15];
let mut acc = *hinted_signature.signature.signature;
for witness in squarings {
    acc = checked_reduction(@acc, @acc, modulus, witness);
}
acc = checked_reduction(@acc, hinted_signature.signature.signature, modulus, r16);
```

This works because each `RSA2048ReductionWitness` is consumed exactly once and only needs `Drop`.

## Public Verifier Flow

`is_valid_rsa2048_signature_assuming_encoded_message` does the following:

1. validate `public_key.modulus`
2. validate `signature.signature`
3. validate `signature.expected_message`
4. reject `N <= 1`
5. reject even `N`
6. reject `signature >= N`
7. reject `expected_message >= N`
8. run the hardcoded `17`-step witness path
9. return `true` only if the final chunks equal `expected_message.chunks`

Return `false` for bad witness data. Do not panic from the public verifier.

## Pitfalls To Avoid

- Do not add RSA as a top-level `pub mod rsa;`; it belongs under `pub mod signatures`.
- Do not handwrite the repeated Horner evaluator in the signature verifier; keep it generated.
- Do not use `mul_mod_p` and `add_mod_p` one-op helpers for per-channel relation checks.
- Do not use dynamic `Array` or `Span` inside the fixed witness structs.
- Do not declare `RSA_TOP_CHUNK_LIMB1_MAX` as `u96`.
- Do not assume `u96` supports ordering comparisons; upcast to `u128` first.
- Do not re-run `check_encoding(modulus)` or `check_encoding(signature)` inside every reduction step.
- Do not force RSA through the curve-based `write_test_file()` helper.
- Do not reuse `CurveID`, `BASE`, or `N_LIMBS` constants from curve-signature builders for RSA widths.
- Do not compare only residues for the final result; compare final chunks to `expected_message.chunks`.
- Do not name the low-level API as if it were full PKCS#1 verification.
